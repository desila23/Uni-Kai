```c
Ricerca_m(A, i, j)
	if (i > j) then return -1
	
	m = [ i+j / 2 ]    // PARTE INFERIORE
	
	if(A(m) > A(m + 1) && A(m) > A(m - 1)) then 
		return m
	if (A(m) < A(m + 1)) then 
		return ricerca_m(A, m+1, j)
	else 
		return ricerca_m(A, i, m-1)


```

#### COSTO

Creo albero -> nodi = log(n)
Costo di ogni nodo = 1
Costo totale? $O(log(n))$


#### Correttezza
Abbiamo quattro casi principali nella ricerca del massimo
1. Il massimo si trova ai margini -> li controllo entrambi e lo estraggo
2. Partendo dal centro, l'elemento che trovo è il massimo -> lo estraggo
3. L'elemento è più piccolo del successivo, vuol dire che sono in un fase "crescente" -> riesco a capire in tempo costante che devo cercare il massimo a DX dell'elemento corrente
4. L'elemento è più grande del successivo, vuol dire che sono in una fase "decrescente" -> riesco a capire in tempo costante devo cercare il massimo a SX dell'elemento corrente

Inoltre, il costo è rispettato in quanto al più vengono analizzati $log(n)$ elementi, quindi
$$O(log(n)) = o(n)$$



#### PSEUDOCODICE (B)
```c
Ordina_Unimodoale(A)
	n = len(A)
	m = Max(A)
	
	if m != -1 && m != A[n] then 
		Inverti_array(A, m, n)
		merge(A, 1, m, n)
```

```c
Inverti_array(A, i, j)
	while(i < j) do
		ausilio = A[i]
		A[i] = A[j]
		A[j] = ausilio
		i++, j--
	return A
```

#### Correttezza
I casi principali da analizzare sono 
1. Se `m = A[n]` allora l'array è già ordinato (prima di m ho solo valori più piccoli e crescenti, dopo m non ho nulla)
2. Se `m != A[n]` allora il massimo si trova in una qualche altra posizione dell'array, allora
	- so che gli elementi prima di m sono ordinati in ordine CRESCENTE e che quelli dopo sono ordinati in ordine DECRESCENTE
	- uso una funzione ausiliaria `Inverti_array`  così da ordinare in modo crescente gli elementi successivi a `m`
	- utilizzo il `merge` per unire i due "array"
		- `array_1` = da 1 a `m`
		- `array_2` = da `m+1` a `n`
